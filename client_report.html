Web Fundamentals Team Project

Client, Lead Developer, and Junior Developer Report

This report reflects on my experience working in three different roles during the team website project. It explains how I communicated, contributed, managed challenges, and developed greater confidence working in a collaborative web-development environment.

Client Report

How did you decide what content and images to use for your site plan?

I began by identifying the purpose of the site and the audience I wanted it to serve. For the Battlefront Fan Hub, I wanted to create a place for Star Wars Battlefront II fans to learn about the game, view community content, and show support for the series. I selected written content that explained the game clearly and used images that supported the science-fiction theme, including characters, maps, battles, and gameplay scenes. I also tried to make sure every image had a purpose instead of using images only as decoration.

How did you communicate with your developer? Were there any issues with communication?

I communicated with my developer by sharing the site plan, page requirements, color palette, image ideas, and the overall visual direction. I also gave feedback after reviewing the website and explained specific changes that I wanted. Communication was generally successful, although there were times when updates took longer than expected or when I needed to explain a design idea more clearly. I learned that detailed written instructions and visual examples reduce confusion.

Did your developer meet your expectations for your site?

Yes. The developer created a site that followed the requested theme and included the planned navigation, pages, images, colors, and content. The final site reflected the cinematic Battlefront style that I wanted. A few details required revisions, such as image sizing, page links, and the hero background, but the developer responded to feedback and improved the final result.

What will you do differently next time you hire a team to do a project for you?

Next time, I would provide a more detailed project brief at the beginning, including exact page deadlines, image dimensions, required file names, and examples of the expected layout. I would also schedule regular progress checks instead of waiting until the end to review the work. This would make it easier to identify problems early and would help the developer understand my expectations from the start.

Lead Developer Report

How did you manage your development team and communicate the tasks for your junior developers?

I managed the team by dividing the website into pages and assigning clear responsibilities to each junior developer. I shared the common header, navigation, footer, colors, fonts, and file structure so that every page would remain consistent. I communicated tasks through messages and explained which content, images, and layout each developer was responsible for. I also reviewed the submitted work and suggested changes when the page did not fully match the site plan.

How did you communicate with your client? Were there any issues?

I communicated with the client by asking questions about the site purpose, desired style, page content, and design priorities. I shared progress and used the client's feedback to make revisions. The main challenge was making sure that broad requests were converted into specific development tasks. When an instruction was not completely clear, I used the site plan and previous feedback to make a reasonable design decision.

How did you handle publishing your client's site and sharing it with the client?

I published the site through GitHub Pages. Before deployment, I checked the navigation links, image paths, file names, and folder structure. After pushing the final changes, I opened the live site and tested each page. I then shared the published URL with the client. When the deployment remained queued and a hero image did not appear, I reviewed the GitHub Pages workflow and corrected the image path before publishing again.

How was your experience with managing your site with GitHub?

GitHub was useful because it gave the team one place to store files, track changes, and publish the website. However, it also required careful attention to branches, commits, pull requests, folder names, and merge conflicts. I experienced a queued deployment and learned that even a successful build can still display errors when paths use the wrong capitalization or location. Overall, GitHub helped me understand how real development teams organize and publish shared projects.

Junior Developer Report

How did you communicate with your team lead? Were there any issues?

I communicated with each team lead by reviewing the assigned site plan, asking about shared design elements, and sending updates as I worked on my page. I also compared my page with the lead developer's files so the navigation, colors, fonts, and footer would remain consistent. In some cases, communication was slower than I expected, so I continued working from the available wireframe and requirements instead of allowing the project to stop.

Did you know what was expected of you for each site?

Most of the expectations were clear because each assignment included a page topic, site plan, wireframe, and shared visual style. When something was unclear, I identified the required content first and then used the existing site design as a guide. I learned that junior developers need to understand both their individual page and the overall website so their work fits naturally with the rest of the project.

Were you able to contribute to your sites?

Yes. I created complete child pages using semantic HTML and CSS, added responsive layouts, selected appropriate icons, improved hero image placement, and matched the shared navigation and footer. I also gave design feedback when I noticed differences between the wireframe and the current website. My contributions helped the teams complete pages that were visually consistent and functional.

Did you run into any issues with GitHub and adding your changes to the sites?

I had to be careful when adding files because another developer could be changing the same repository. The main risks were editing the wrong branch, placing files in an incorrect folder, or overwriting shared CSS. I learned to pull the newest changes before editing, keep my work focused on the assigned files, write clear commit messages, and verify the site after merging my changes.

General Experience and Reflections

What was the most challenging part of this project for you? Were you able to overcome it?

The most challenging part was coordinating work across different roles while keeping each website consistent. A small difference in file paths, colors, navigation, or image sizes could make the pages feel disconnected. Publishing through GitHub also created challenges when a deployment stayed queued or an image failed to load. I overcame these problems by checking the code carefully, comparing file paths, testing the live site, and making revisions one issue at a time.

What did you enjoy most about this project?

I enjoyed seeing separate ideas and pages become complete websites. It was satisfying to move beyond a simple individual assignment and work with site plans, clients, team roles, GitHub, and live deployment. I also enjoyed improving visual details such as hero sections, responsive grids, icons, and consistent branding.

What would you do differently if you could do the project again?

I would establish the shared file structure and style guide before anyone began coding. I would create a checklist for navigation, responsive design, accessibility, image paths, and deployment. I would also communicate progress more consistently and test each page immediately after it was merged rather than waiting until the end.

Do you feel more confident in working on real-world projects after this experience? Explain.

Yes. This project helped me understand that real-world web development requires more than writing HTML and CSS. It also requires communication, planning, version control, feedback, troubleshooting, and responsibility for a final product. I now feel more prepared to receive requirements, work with a team, contribute code through GitHub, and publish a website that another person will actually use.
